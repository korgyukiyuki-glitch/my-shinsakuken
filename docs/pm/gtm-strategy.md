# 上市戦略 (GTM Strategy): マイ診察券

> 作成日: 2026年3月8日
> ステータス: App Store審査待ち → 承認後即ローンチ
> 予算: 個人開発のため実質ゼロ〜月額数千円

---

## 1. エグゼクティブサマリー

マイ診察券は「患者が自分で使える唯一のデジタル診察券」として、ゼロ予算で日本市場にローンチする。初期はASO（App Store最適化）と口コミを中心とした自然獲得に集中し、勤務先医院をテスト拠点として実績を作る。ビーチヘッド・セグメントは「複数医院に通院する40-60代」。90日間で累計1,000DLを目標とし、D30リテンション40%以上を達成して初期PMFシグナルを確認する。

## 2. ビーチヘッド・セグメント分析

### 2.1 候補セグメント評価

| # | セグメント | ペインの深さ | 勝てるか | 紹介連鎖 | アクセス性 | **総合** |
|---|-----------|:-----------:|:-------:|:-------:|:---------:|:-------:|
| A | 複数科通院の中高年（40-60代） | ★★★ | ★★★ | ★★ | ★★ | **10** |
| B | 子供の通院管理をする親（30-40代） | ★★★ | ★★★ | ★★★ | ★★ | **11** |
| C | 歯科に定期通院する若年層（20-30代） | ★ | ★★ | ★★ | ★★★ | **8** |
| D | 高齢の親を代理管理する子世代（40-50代） | ★★ | ★★★ | ★ | ★ | **7** |

### 2.2 評価基準の詳細

**ペインの深さ**: 既存の解決策（紙の診察券）に対する不満度
- A: 5枚以上の診察券を持ち歩く。受付で探す手間。紛失リスク → ★★★
- B: 自分+子供で多数の診察券。予約日も忘れがち → ★★★
- C: 歯科1院のみが多い。ペインが浅い → ★
- D: ペインはあるが頻度が低い（付き添い時のみ） → ★★

**勝てるか**: 競合に対するマイ診察券の優位性
- A: 競合は全て医院主導型で全診療科非対応。圧倒的に勝てる → ★★★
- B: 同上。家族管理はPhase 1.5で対応予定 → ★★★
- C: Apotool（歯科特化）が強い。単一医院だと差別化薄い → ★★
- D: 競合なし。ただしニッチすぎる → ★★★

**紹介連鎖**: 1人のユーザーが他のユーザーを呼ぶ力
- A: 待合室で「それ何のアプリ？」と話題に → ★★
- B: ママ友ネットワーク。LINEグループでの情報共有が活発 → ★★★
- C: SNSでのシェアが期待できる → ★★
- D: 介護コミュニティへの波及は限定的 → ★

**アクセス性**: 個人開発者がリーチできるか
- A: 勤務先医院の患者が直接ターゲット → ★★
- B: SNS（Instagram, Twitter）でリーチ可能 → ★★
- C: デジタルネイティブ。ASO・SNSで最もリーチしやすい → ★★★
- D: リーチ手段が限定的 → ★

### 2.3 推奨ビーチヘッド

**セグメントB（子育て世代）を最優先ビーチヘッドとする**

理由:
1. ペインが深く、複数の子供×複数の診療科で枚数が多い
2. ママ友ネットワークによる自然な口コミ拡散力が最も強い
3. デジタルリテラシーが高く、アプリ導入への心理的ハードルが低い
4. SNS（Instagram、X）でのリーチが容易
5. 「子供の診察券管理」は競合が全くカバーしていない独自ポジション

**セカンダリ: セグメントA（複数科通院の中高年）**
- ペインは最も深いが、自然な口コミ拡散力がBに劣る
- 勤務先医院の患者として直接アプローチ可能

## 3. マーケティングチャネル戦略

### 3.1 ASO（App Store Optimization）— **最重要チャネル**

App Storeユーザーの約70%がキーワード検索でアプリを発見し、DLの約65%が検索経由。予算ゼロの個人開発では最もROIが高いチャネル。

#### キーワード戦略

**現在のキーワード（100文字）**:
`診察券,病院,医院,歯科,歯医者,内科,眼科,皮膚科,予約,リマインダー,お薬手帳,通院,患者番号,QRコード,診療`

**改善案**:
```
診察券,病院,医院,歯科,内科,眼科,皮膚科,予約,通院,お薬,患者番号,QR,リマインダー,受付,カード管理
```

改善ポイント:
- 「歯医者」を削除（「歯科」で十分カバー）→ 文字数を他に回す
- 「お薬手帳」→「お薬」に短縮（部分一致で「お薬手帳」もカバー）
- 「QRコード」→「QR」に短縮
- 「診療」→「受付」に変更（ユーザーの行動に近いキーワード）
- 「カード管理」を追加（「診察券 管理」の検索にヒット）

#### スクリーンショット最適化

| 順番 | 内容 | メッセージ |
|------|------|-----------|
| 1 | ホーム画面（複数医院表示） | 「全ての診察券を、ひとつに」 |
| 2 | 診察券詳細（患者番号大画面表示） | 「受付で見せるだけ」 |
| 3 | QRコード表示 | 「QRコードですぐ受付」 |
| 4 | 予約一覧（リマインダー付き） | 「予約を忘れない」 |
| 5 | お薬管理 | 「お薬もまとめて管理」 |
| 6 | 設定画面（生体認証） | 「Face ID / Touch IDで安心」 |

#### カテゴリ戦略

- **プライマリ**: メディカル（Medical）— 競合が少なく上位表示されやすい
- **セカンダリ**: ヘルスケア／フィットネス — 検索ボリュームが大きい

#### レビュー獲得戦略

1. **タイミング**: 2枚目の診察券登録完了時にレビュー依頼（Expo StoreReview API）
   - 理由: アプリの価値を実感した直後が最もポジティブな評価を得やすい
2. **頻度**: 30日に1回まで（Apple ガイドライン準拠）
3. **目標**: 公開後3ヶ月で50件以上のレビュー、平均4.5以上

### 3.4 口コミ・バイラル

#### 勤務先医院でのシードユーザー獲得（最重要）

1. 受付にアプリ紹介カードを設置（許可取得済み前提）
2. 患者に「診察券アプリ、よかったら使ってみてください」と案内
3. 初期50人のコアユーザーを勤務先で獲得
4. フィードバックを直接収集しアプリ改善に反映

#### 自然な口コミ発生メカニズム

```
患者がアプリで受付
    ↓
他の患者が「それ何のアプリ？」と興味
    ↓
「マイ診察券っていうアプリだよ」
    ↓
新しいユーザーがダウンロード
    ↓
その人も受付で使う → ループ
```

→ 受付という「公の場」で使うアプリだからこそ、自然な口コミが発生しやすい

## 5. グロースループ設計

### 5.1 プライマリループ: 受付場面での自然発見

```
┌──────────────────────────────────────────┐
│                                          │
│  ユーザーが受付でアプリを使う             │
│         ↓                                │
│  他の患者・受付スタッフが目撃            │
│         ↓                                │
│  「何のアプリ？」と会話が発生            │
│         ↓                                │
│  App Storeで検索・ダウンロード           │
│         ↓                                │
│  新ユーザーが自分の医院を登録            │
│         ↓                                │
│  次回の受付でアプリを使う ──→ ループ     │
│                                          │
└──────────────────────────────────────────┘
```

**このループの強み**:
- 広告費ゼロで発動する
- 「受付」という繰り返しの行動に紐づいている（毎回の通院で発火）
- 医療機関の待合室は「退屈」な場所で、他人の行動に目が行きやすい

### 5.2 セカンダリループ: ボトムアップ医院導入

```
┌──────────────────────────────────────────┐
│                                          │
│  患者が受付で「このアプリ使ってます」     │
│         ↓                                │
│  受付スタッフが複数の患者から同じ声を聞く │
│         ↓                                │
│  医院長に「こんなアプリがあるらしい」報告 │
│         ↓                                │
│  医院がPhase 4のWebアプリ導入を検討      │
│         ↓                                │
│  医院が患者に「マイ診察券使えます」案内   │
│         ↓                                │
│  その医院の全患者に広がる ──→ ループ     │
│                                          │
└──────────────────────────────────────────┘
```

**このループの強み**:
- 患者→医院→患者のフライホイール効果
- Phase 4の収益化に直結
- 競合（トップダウン型営業）とは真逆のアプローチ

## 8. リスクと対策

### 8.1 事業リスク

| リスク | 影響度 | 発生確率 | 対策 |
|--------|:------:|:--------:|------|
| **Apotoolが複数医院対応を実装** | 極高 | 中 | スピード重視でユーザー基盤を先行構築。プライバシー・UX・全診療科で差別化維持 |
| **App Store審査リジェクト** | 高 | 低 | 医療判断機能なし・データ収集なしで審査対策済み。リジェクト時は即対応 |
| **ユーザー獲得が伸びない** | 高 | 中 | SEO記事・SNS強化。ASO継続改善。勤務先での地道な口コミ |
| **大手IT企業の参入** | 極高 | 低 | ニッチ市場への深掘り。医療特化のドメイン知識で差別化 |

### 8.2 プロダクトリスク

| リスク | 影響度 | 発生確率 | 対策 |
|--------|:------:|:--------:|------|
| **リテンション率が低い** | 高 | 中 | 予約リマインダーによる定期的な再エンゲージメント |
| **1医院しか登録されない** | 中 | 中 | オンボーディングで「2医院以上の登録」を促す設計 |
| **医院の受付がアプリに否定的** | 中 | 低 | 患者番号の大画面表示で紙の診察券と同じように目視確認可能 |

### 8.3 ヘルスケアアプリ特有のリスク

| リスク | 対策 |
|--------|------|
| アンインストール率が高い（ヘルスケアアプリは全ジャンル4位） | 初回起動からの診察券登録までのステップを最小化。プッシュ通知で再エンゲージ |
| プライバシー懸念による敬遠 | ローカル保存・ゼロデータ収集を前面に訴求 |

| 項目 | 競合（Apotool等） | マイ診察券 |
